# 글 타입: Devlog (개발기)

## 용도
개발 프로젝트의 설계, 구현, 운영 과정을 기록하는 글.
"왜 이 선택을 했는지"에 집중한다. 코드 나열이 아니라 의사결정 과정을 서술.

## 시리즈 번호
- 시리즈물일 경우 제목 앞에 `(01)`, `(02)` 형식으로 번호 붙인다
- 예: `(01) OpenClaw를 사용한 개인 홈페이지 개발기 — 설계`

## 메타 섹션 (독자에게 보임)

TL;DR 바로 다음, 본문 전에 배치. blockquote로 감싼다:

```markdown
## TL;DR
- ...

> **주요 목표**: {이 글에서 다루는 개발의 핵심 목표 한 줄}
>
> **Human**: {유저가 직접 한 것 — 방향 설정, 의사결정, 기획, 리뷰, 디자인 판단 등}
>
> **AI**: {AI가 기여한 것 — 코드 구현, 리서치, 초안 작성, 리팩토링, 테스트 등}

---

## 1) 배경/동기
```

## 본문 구조

```
1) 배경/동기 — 왜 이걸 만들게 됐는가
2~N) 개발 과정 — 각 섹션은 하나의 결정/구현 단위
    - 무엇을 했는지 (결과)
    - 왜 이 방향으로 갔는지 (이유)
    - 시행착오가 있었다면 (삽질 & 해결)
N+1) 교훈 / 레슨런
N+2) 더 생각해볼 점 & 보완/개선할 점
N+3) 다음 단계
```

### 교훈 섹션 (테이블 권장)

```markdown
| 원칙 | 왜 |
|------|-----|
| 문서가 곧 실행 | 에이전트가 읽을 수 있어야 자동화가 된다 |
| DONE = REVIEW:done | "코드 짰다"와 "끝났다"는 다른 말이다 |
```

## 마무리 섹션: 더 생각해볼 점 & 보완/개선할 점

교훈 섹션 다음, "다음 단계" 전에 배치. 글에서 다룬 접근법의 한계나 열린 질문을 솔직하게 적는다.

```markdown
## N+1) 더 생각해볼 점

- **{주제 1}**: {현재 방식의 한계 또는 아직 검증 안 된 부분}
- **{주제 2}**: {다른 접근이 더 나았을 수 있는 지점}
- **{주제 3}**: {스케일, 유지보수, 장기적 관점에서 열린 질문}
```

### 작성 팁
- "완벽했다"로 끝내지 않는다. 열린 질문이 있어야 글에 깊이가 생긴다.
- 자기 비판이 아니라 **발전 방향 제시**: "X가 부족했다"보다 "X를 Y로 바꾸면 더 나을 수 있다"
- 독자가 자기 상황에 적용할 때 고려할 것도 여기에 넣는다
- 2~4개가 적당. 너무 많으면 본문이 부실해 보인다.

## Phase 1에서 물어볼 것

1. **주요 목표**: 이 개발에서 달성하려던 핵심 목표가 뭐야?
2. **Human 역할**: 네가 직접 한 건 뭐야? (방향 설정, 리뷰, 디자인 결정 등)
3. **AI 역할**: AI가 기여한 부분은? (구현, 리서치, 초안 등)
4. **핵심 결정**: 가장 중요했던 기술/설계 결정 1~3개는?
5. **시행착오**: 삽질한 거 있으면? (있으면 좋은 콘텐츠가 됨)
6. **교훈**: 이 과정에서 배운 것 or 다음에 다르게 할 것?
