# 글 타입: Tech (기술 해설)

## 용도
특정 기술, 아키텍처, 패턴, 도구를 설명하는 글.
개발기와 다르게 "시간 순서"가 아니라 "개념 → 실제 → 주의사항" 순서.

## 메타 섹션 (독자에게 보임)

TL;DR 다음에 배치:

```markdown
## TL;DR
- ...

> **다루는 것**: {기술/개념 한 줄}
>
> **대상 독자**: {예: "Astro를 써본 적 있고, Cloudflare Workers에 관심 있는 개발자"}
>
> **선행 지식**: {예: "기본적인 SSG 개념, REST API"}

---

## 1) 문제/동기
```

## 본문 구조

```
1) 문제/동기 — 왜 이 기술이 필요한가, 어떤 문제를 해결하는가
2) 핵심 개념 — 기술의 핵심을 2~3가지로 압축
3) 실제 구현/예시 — 코드, 설정, 다이어그램으로 보여주기
4) 주의사항 & 트레이드오프 — 한계, 대안, "이것만은 조심"
5) 더 생각해볼 점 — 열린 질문, 개선 여지, 미탐색 영역
6) 정리 — 핵심 요약 + 다음에 읽을 것/해볼 것
```

### 핵심 개념 섹션 팁
- 한 섹션에 개념 1개씩
- 개념마다 "한 줄 정의 → 왜 중요한지 → 어떻게 동작하는지" 흐름
- 다이어그램(mermaid) 적극 사용

### 코드 예시 팁
- 전체 파일 복사가 아니라 핵심 부분만 발췌
- 앞뒤에 설명 반드시 달기 ("이 코드에서 중요한 건 X줄이다")
- 실행 가능한 코드 선호 (복사해서 바로 돌릴 수 있게)

## 마무리 섹션: 더 생각해볼 점

"주의사항" 다음, "정리" 전에 배치. 글에서 다루지 못했거나 깊이 파지 못한 영역을 짚는다.

```markdown
## 5) 더 생각해볼 점

- **{미탐색 영역}**: {이 글에서 다루지 않았지만 관련된 주제}
- **{스케일 관점}**: {규모가 커지면 달라질 수 있는 부분}
- **{대안 접근}**: {다른 방식으로 풀 수 있는 가능성}
```

### 작성 팁
- 글의 한계를 인정하면 신뢰도가 올라간다
- "이 글에서는 X를 다루지 않았다. Y 상황이라면 Z를 고려해야 한다" 식으로
- 독자가 자기 맥락에서 추가로 조사할 키워드를 제공하는 역할

## Phase 1에서 물어볼 것

1. **다루는 기술/개념**: 정확히 뭘 설명하고 싶어?
2. **독자 레벨**: 초보? 중급? 이 분야 경험자?
3. **실용 예시**: 실제 코드/설정 예시를 넣을 거야?
4. **핵심 메시지**: 독자가 이 글에서 가져가야 할 1가지는?
5. **비교/대안**: 다른 접근법이나 대안도 다룰 거야?
