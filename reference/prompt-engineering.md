# Agent/Skill용 프롬프트 엔지니어링

## CLAUDE.md 작성 원칙

### 1. 명확한 역할 부여
```markdown
You are a code reviewer specializing in security.
Focus on: SQL injection, XSS, CSRF, auth bypass.
```

### 2. 구체적인 행동 지시
```markdown
# 나쁜 예
코드를 잘 작성해주세요.

# 좋은 예
- 함수는 단일 책임 원칙을 따를 것
- 에러 핸들링은 호출자에게 위임
- 타입은 any를 사용하지 않을 것
```

### 3. 출력 형식 지정
```markdown
## 응답 형식
1. 요약 (1-2줄)
2. 변경 파일 목록
3. 각 변경에 대한 설명
```

## Skill 프롬프트 패턴

### Chain of Thought 유도
```markdown
## 단계
1. 먼저 현재 상태를 분석
2. 문제점을 식별
3. 해결 방안을 3가지 제시
4. 최적의 방안을 선택하여 구현
```

### 도구 사용 가이드
```markdown
## 사용할 도구
- Grep: 코드베이스에서 패턴 검색
- Read: 파일 내용 확인
- Edit: 코드 수정
- Bash: 테스트 실행

## 주의사항
- Edit 전에 반드시 Read로 파일 확인
- Bash로 rm 명령 사용 금지
```

### 가드레일 설정
```markdown
## 하지 말 것
- 사용자 확인 없이 파일 삭제하지 말 것
- package.json의 의존성을 임의로 변경하지 말 것
- main 브랜치에 직접 push하지 말 것

## 불확실할 때
- AskUserQuestion으로 사용자에게 확인
```

### 인터랙티브 워크플로우
```markdown
## 워크플로우
1. 사용자에게 주제를 물어봄
2. 초안 생성 후 사용자 리뷰 요청
3. 피드백 반영
4. 최종 확인 후 저장

각 단계에서 사용자의 명시적 승인을 받을 것.
```

## 서브에이전트 프롬프트 팁

### 명확한 범위 설정
```
"이 프로젝트의 src/ 디렉토리에서 모든 API 엔드포인트를 찾아서
각각의 HTTP 메서드, 경로, 인증 요구사항을 정리해줘.
결과는 마크다운 테이블로."
```

### 결과 형식 지정
```
"검색 결과를 다음 형식으로 반환:
- 파일 경로
- 관련 코드 스니펫 (10줄 이내)
- 왜 관련있는지 한 줄 설명"
```

## 프롬프트 디버깅

- `--verbose`로 실제 전송되는 프롬프트 확인
- 스킬이 트리거되지 않으면 description을 더 구체적으로
- 예상과 다른 동작이면 "하지 말 것" 목록 추가
- 복잡한 지시는 번호 매긴 단계로 분리
