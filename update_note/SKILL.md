# Update Note - 세션 작업 내용 기록

현재 세션에서 작업한 주요 내용을 `update_note.md` 파일에 요약하여 기록합니다.

## 작업 순서

1. **현재 워크스페이스 확인**
   - 현재 작업 중인 디렉토리의 경로를 확인합니다.

2. **update_note.md 파일 확인 및 생성**
   - 워크스페이스에 `update_note.md` 파일이 있는지 확인합니다.
   - 파일이 없으면 새로 생성합니다.
   - 파일이 있으면 기존 내용을 읽어옵니다.

3. **세션 작업 내용 요약**
   현재 세션에서 다음 내용들을 요약하여 작성합니다:

   - **날짜 및 시간**: 현재 날짜와 시간을 기록합니다.
   - **주요 작업 내용**: 어떤 파일을 수정했고, 무엇을 추가/변경/삭제했는지
   - **문제 해결**: 발생했던 이슈와 해결 방법
   - **중요한 결정사항**:
     - 왜 특정 방식을 선택했는지
     - 어떤 대안들이 있었는지
     - 향후 고려사항
   - **코드 변경 사항**:
     - 추가된 함수/클래스/모듈
     - 변경된 로직
     - 리팩토링 내용
   - **테스트 및 검증**: 실행한 테스트나 확인 절차
   - **다음 할 일**: 아직 완료되지 않은 작업이나 후속 작업

4. **기록 형식**
   ```markdown
   ## [날짜] - [작업 제목]

   ### 작업 내용
   - 핵심 작업 내용 1
   - 핵심 작업 내용 2

   ### 주요 변경사항
   - `파일경로`: 변경 내용 설명

   ### 중요 결정사항
   - 결정 내용과 이유

   ### 문제 해결
   - 문제: [문제 설명]
   - 해결: [해결 방법]

   ### 참고사항
   - 추가 정보나 유용한 팁

   ### 다음 단계
   - [ ] 할 일 1
   - [ ] 할 일 2

   ---
   ```

5. **파일 업데이트**
   - 기존 파일이 있으면 새 내용을 맨 위에 추가합니다 (최신 내용이 위로).
   - 가독성을 위해 항목 간 구분선(`---`)을 사용합니다.

6. **Git Commit 제안** (워크스페이스가 git 저장소인 경우)

   변경사항을 분석하여 논리적인 커밋 단위로 쪼개서 제안합니다:

   a. **Git 상태 확인**
      - `git status`로 변경된 파일 목록 확인
      - `git diff`로 변경 내용 확인
      - staged/unstaged 파일 구분

   b. **변경사항 그룹핑**
      변경된 파일들을 다음 기준으로 논리적 단위로 그룹화:
      - **기능별**: 같은 기능/피처 관련 변경사항
      - **파일 타입별**: 코드, 테스트, 문서, 설정 파일
      - **의존성**: 함께 커밋되어야 하는 연관된 변경사항
      - **범위**: 작은 단위로 쪼갤수록 좋음 (atomic commits)

   c. **커밋 메시지 제안**
      각 그룹별로 다음 형식의 커밋 메시지 작성:
      ```
      [타입]: 간결한 제목 (50자 이내)

      - 변경사항 설명 1
      - 변경사항 설명 2

      이유: [왜 이 변경이 필요했는지]
      ```

      타입 예시: feat, fix, refactor, docs, test, chore, style, perf

   d. **사용자 승인 요청**
      - 각 커밋 그룹과 메시지를 명확하게 제시
      - "이렇게 커밋할까요?" 라고 물어보기
      - 사용자가 수정을 원하면 조정 가능
      - 승인 받으면 순차적으로 커밋 실행

   **커밋 실행 예시:**
   ```bash
   # 첫 번째 커밋 그룹
   git add [관련 파일들]
   git commit -m "커밋 메시지"

   # 두 번째 커밋 그룹
   git add [관련 파일들]
   git commit -m "커밋 메시지"
   ```

## 사용 예시

```bash
# 전체 세션 내용 요약
/update_note

# 특정 주제에 집중하여 요약
/update_note authentication refactoring
```

## 주의사항

- 코드의 세부 구현보다는 **"왜"** 그렇게 했는지에 집중합니다.
- 향후 참고할 때 도움이 될 만한 **컨텍스트**를 충분히 포함합니다.
- 민감한 정보(API 키, 비밀번호 등)는 절대 기록하지 않습니다.
- 파일 경로는 워크스페이스 루트 기준 상대 경로로 작성합니다.

## 인수 사용

`$ARGUMENTS`가 제공되면 해당 주제에 집중하여 요약을 작성합니다.
예: `/update_note database migration` → 데이터베이스 마이그레이션 관련 작업에 초점
