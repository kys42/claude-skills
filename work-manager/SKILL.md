# Work Manager - 업무 워크플로우 매니저

업무 작업 흐름을 단계별로 관리하는 워크플로우 매니저입니다.
사용자가 작업을 요청하면 아래 단계를 **순서대로** 진행합니다. (Step 0~9)

---

## 자동 상태 감지 (작업 지시 없이 실행 시)

사용자가 구체적인 작업 지시 없이 이 스킬을 실행하면, 현재 진행 상황을 자동으로 파악하여 다음 액션을 제안합니다.

### 감지 절차
1. `~/.claude/skills/work-manager/scripts/detect-status.sh [프로젝트경로]` 스크립트를 실행하여 git/GitHub 상태를 한번에 수집합니다
2. 세션 히스토리를 참고하여 이전 대화에서 어떤 Step까지 진행했는지 확인합니다
3. 수집된 정보를 종합하여 현재 단계를 판단합니다

### 상태 판단 기준

| 감지 신호 | 추정 단계 | 다음 액션 제안 |
|-----------|----------|---------------|
| main/develop 브랜치 + 변경 없음 + 열린 이슈 없음 | 시작 전 | "새 작업을 시작할까요? (Step 1: 플랜 수립)" |
| main/develop 브랜치 + 열린 이슈 있음 | Step 2 완료 | "이슈 #{N}에 대해 브랜치를 생성하고 구현을 시작할까요? (Step 3)" |
| feature 브랜치 + 변경 없음 + 커밋 없음 | Step 3 시작 | "브랜치가 생성되어 있습니다. 구현을 시작할까요? (Step 3)" |
| feature 브랜치 + 변경 있음 (uncommitted) | Step 3 진행중 | "커밋되지 않은 변경이 있습니다. 테스트 후 커밋할까요? (Step 4)" |
| feature 브랜치 + 커밋 있음 + PR 없음 | Step 4 완료 | "커밋이 준비되어 있습니다. PR을 생성할까요? (Step 5)" |
| feature 브랜치 + 열린 PR 있음 + 리뷰 전 | Step 5 완료 | "PR #{N}이 생성되어 있습니다. 코드 리뷰를 진행할까요? (Step 6)" |
| feature 브랜치 + PR 머지 완료 + 열린 이슈 있음 | Step 7 완료 | "PR이 머지되었습니다. 관련 이슈를 닫고 UPDATE_NOTE를 작성할까요? (Step 7.5)" |
| feature 브랜치 + PR 머지 완료 + 이슈 닫힘 | Step 7.5 완료 | "이슈가 닫혀있습니다. UPDATE_NOTE를 작성할까요? (Step 8)" |

### 제안 형식
상태 감지 후 사용자에게 아래 형식으로 보고합니다:

```
📍 현재 상태
- 프로젝트: {project-name}
- 브랜치: {branch}
- 추정 단계: Step {N} ({단계명}) 완료

➡️ 다음 액션: Step {N+1} ({단계명})
  {구체적인 다음 액션 설명}

이대로 진행할까요?
```

## 프로젝트 컨텍스트 시스템

프로젝트별 지식 저장소를 관리합니다.

- **컨텍스트 위치**: `~/.claude/skills/work-manager/projects/{project-name}/`
- **프로젝트 이름**: 리포지토리 이름 기본 사용 (사용자 변경 가능)
- **템플릿**: `~/.claude/skills/work-manager/templates/`

### 컨텍스트 구조
```
projects/{project-name}/
├── INDEX.md              # 문서 네비게이션 허브 (가장 먼저 읽을 것)
├── OVERVIEW.md           # 시스템 전체 개요
├── ops/
│   └── OPS.md            # 배포, QA, 테스트 전략
└── features/             # 기능별 상세 문서
    └── {feature-name}/
        └── README.md
```

## 워크플로우 단계

### Step 0. 프로젝트 컨텍스트 로드
- 현재 작업 중인 프로젝트의 컨텍스트가 존재하는지 확인합니다
  - 경로: `~/.claude/skills/work-manager/projects/{project-name}/`
- **컨텍스트가 있으면**:
  1. `INDEX.md`를 읽어 문서 맵을 파악합니다
  2. `OVERVIEW.md`를 읽어 프로젝트 전체 그림을 파악합니다
  3. 작업 대상 기능의 `features/{name}/README.md`가 있으면 함께 읽습니다
  4. 배포/운영 관련 작업이면 `ops/` 하위 문서도 읽습니다
- **컨텍스트가 없으면**:
  - 사용자에게 프로젝트 컨텍스트 초기화 여부를 확인합니다
  - 승인 시 `templates/` 디렉토리의 템플릿을 복사하여 초기 구조를 생성합니다
- 이 단계는 사용자 확인 없이 자동으로 진행합니다 (초기화 제외)

### Step 1. 작업 플랜 수립
- `EnterPlanMode`를 사용하여 구현 계획을 수립합니다
- 코드베이스를 탐색하여 수정 대상 파일과 변경 내용을 구체적으로 정리합니다
- 플랜에는 다음을 포함합니다:
  - 작업 배경 및 목표
  - 변경 대상 파일 목록
  - 각 파일별 수정 내용 요약
  - 검증 방법 (테스트, 수동 확인 등)
- **사용자 확인**: `ExitPlanMode`로 플랜 승인 후 다음 단계 진행

### Step 2. GitHub 이슈 생성
- 승인된 플랜을 기반으로 GitHub 이슈를 생성합니다
- 이슈 제목은 간결하고 명확하게 작성합니다
- 이슈 본문에는 플랜에서 정리한 작업 배경/목표/변경 범위를 포함합니다
- `gh issue create` 명령어를 사용합니다
- 생성된 이슈 번호를 기록합니다 (이후 PR에서 연결)

### Step 3. 브랜치 생성 및 구현
- 현재 브랜치에서 작업하거나, 필요시 새 브랜치를 생성합니다
  - 브랜치명은 사용자에게 확인받습니다
  - 새 브랜치 생성 시 `git worktree`를 사용합니다:
    1. `git worktree add ../<워크트리-디렉토리명> -b <브랜치명>` 으로 워크트리 생성
    2. 이후 구현 작업은 해당 워크트리 디렉토리에서 수행합니다
    3. 작업 완료 및 PR 머지 후 `git worktree remove ../<워크트리-디렉토리명>`으로 정리합니다
  - 워크트리 디렉토리명은 브랜치명과 동일하게 사용합니다 (예: `feature/add-login` → `../feature/add-login`)
- 승인된 플랜에 따라 코드를 구현합니다

### Step 4. 테스트 코드 작성
- 에이전트가 변경 범위와 복잡도를 보고 테스트 코드 필요 여부를 판단합니다
- **작성이 필요한 경우**:
  - 새 함수/클래스/엔드포인트 추가
  - 기존 로직 변경으로 사이드이펙트 가능성이 있는 경우
  - 복잡한 분기/엣지 케이스가 있는 경우
- **건너뛰는 경우**:
  - 단순 설정 변경, 문서 수정, 스타일 변경 등
  - 프로젝트에 테스트 프레임워크가 없는 경우
- 기존 테스트 패턴(위치, 네이밍, 프레임워크)을 따라 작성합니다
- 작성한 테스트가 통과하는지 확인 후 다음 단계로 진행합니다

### Step 5. 테스트 및 품질 검증
- 구현 완료 후 커밋 전에 품질 검증을 수행합니다
- 프로젝트에 설정된 테스트/린트 명령어를 실행합니다 (예: `make test`, `make lint`, `make typecheck`)
- 실패 시 수정하고 다시 검증합니다
- 모든 검증 통과 후 커밋합니다 (commit message는 변경 내용을 자세하게 기술)
- 원격에 push합니다

### Step 6. PR 생성 및 이슈 연결
- `gh pr create`로 PR을 생성합니다
- PR 본문에 `Closes #이슈번호`를 포함하여 Step 2의 이슈와 연결합니다
- PR 내용 구성:
  - **Summary**: 변경 사항 요약 (bullet point)
  - **주요 의사결정 & 특이점**: 구현 과정에서 내린 기술적 판단, 트레이드오프, 주의할 점, 일반적이지 않은 접근 방식 등을 기재
  - **변경 파일**: 파일별 변경 내용
  - **Test plan**: 검증 방법 체크리스트
- base 브랜치는 사용자에게 확인받습니다
- CI가 있으면 PR 생성 후 CI 결과도 확인합니다

### Step 7. 코드 리뷰 (서브에이전트)
- `Task` tool로 서브에이전트를 생성하여 PR의 변경 사항을 리뷰합니다
- **리뷰 범위**: PR의 diff(변경된 코드)에 집중합니다. `gh pr diff`로 실제 변경분만 확인합니다
- 리뷰 관점:
  - 코드 품질 및 일관성
  - 버그 가능성 및 엣지 케이스
  - 기존 코드와의 호환성
  - 보안 취약점 (OWASP Top 10)
  - 누락된 예외 처리
- **기존 코드 이슈 분리**: PR 변경분이 아닌 기존 코드에서 이슈가 발견된 경우:
  - 리뷰 본문에 `+@ 기존 코드 이슈` 섹션으로 별도 분리하여 기재합니다
  - 사용자에게 해당 이슈들을 GitHub Issue로 등록할지 확인합니다
  - 등록 시 영향 브랜치(develop, main 등)를 이슈 본문에 명시합니다
  - PR 리뷰 자체의 승인/반려 판단에는 기존 코드 이슈를 포함하지 않습니다
- 리뷰 결과를 사용자에게 보고합니다
- 수정이 필요하면 반영 후 Step 5(테스트 검증)부터 다시 진행합니다
- 리뷰 완료 후 PR을 머지합니다 (머지 방식은 사용자에게 확인)

### Step 7.5. 관련 이슈 닫기
- PR 머지 후, Step 2에서 생성한 이슈가 자동으로 닫히지 않았다면 `gh issue close`로 닫습니다
- PR 본문에 `Closes #N`이 포함되어 있어도 base 브랜치가 기본 브랜치(main/master)가 아니면 자동 닫힘이 안 되므로 수동 확인 필요
- 이슈 닫을 때 완료 코멘트를 남깁니다 (어떤 PR로 해결되었는지 명시)
- **사용자 확인 없이 자동으로 진행합니다**

### Step 8. 작업 기록 (UPDATE_NOTE)
- `/update_note` 스킬을 실행하여 UPDATE_NOTE.md에 작업 내용을 기록합니다
- 이번 워크플로우에서 수행한 작업 내용을 간결하게 정리합니다
- **사용자 확인 없이 자동으로 진행합니다**

### Step 9. 프로젝트 컨텍스트 업데이트
- 이번 작업에서 **의미 있는 변경**이 있을 때만 수행합니다 (에이전트 판단):
  - 새 기능 추가 → `features/{name}/` 폴더 생성 + README.md 작성
  - 아키텍처 변경 → `OVERVIEW.md` 업데이트
  - 배포/QA 변경 → `ops/OPS.md` 업데이트
  - 기존 기능 수정 → 해당 `features/{name}/README.md` 변경 이력 추가
- `INDEX.md`의 문서 목록과 마지막 업데이트 날짜를 갱신합니다
- **사용자 확인**: 업데이트할 내용을 요약 후 승인받고 진행합니다
- 의미 있는 변경이 아니면 이 단계를 건너뜁니다

## 진행 규칙

1. **단계 표시**: 각 단계 시작 시 `**[Step N/9] 단계명**` 형식으로 알립니다 (Step 0은 자동 진행)
2. **단계 완료 보고**: 각 단계가 끝나면 주요 변경점/결과를 간결하게 알립니다
3. **자동 진행**: 사용자의 의사결정이 필요 없으면 멈추지 않고 바로 다음 단계로 진행합니다. 어떤 단계에서 멈출지는 고정이 아니라 **상황에 따라 에이전트가 판단**합니다. (예: 플랜 승인, 리뷰 결과 등은 보통 확인이 필요하지만, 단순 작업이면 묶어서 진행해도 됨)
4. **건너뛰기**: 이미 완료된 단계가 있으면 사용자 확인 후 건너뜁니다 (예: 이슈가 이미 존재)
5. **중단/재개**: 사용자가 중단을 요청하면 현재까지 진행 상황을 요약합니다
6. **최종 요약**: 모든 단계 완료 후 아래 내용을 정리합니다:
   - 이슈 번호 + 링크
   - PR 번호 + 링크
   - 브랜치명
   - 변경 파일 목록
   - 테스트/린트 결과
   - 리뷰 결과 요약
   - UPDATE_NOTE 작성 완료 여부
   - 컨텍스트 업데이트 여부
